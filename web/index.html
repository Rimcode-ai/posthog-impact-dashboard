Methodology click to collapse

Window: merged PRs in the last 90 days. Engineers with fewer than 3 merged PRs are excluded (signal too noisy). Bots and the posthog service account are excluded.

The 5 signals (each z-scored across the eligible pool, then weighted):

  • Surviving code (25%) — sum of additions per merged PR, capped at 2,000 lines/PR to prevent one big PR from dominating. PRs whose title is reverted by a later "Revert ..." PR receive 40% credit.
  • Review leverage (25%) — number of PRs where the engineer left an approving review with at least one comment ≥25 chars (filters out "lgtm" / "👍").
  • Cross-area reach (15%) — number of distinct top-level directories touched across their merged PRs. Captures breadth of trust.
  • Incident work (20%) — merged PRs that closed an issue (or were themselves) labeled bug / incident / p0 / p1 / sev / hotfix / regression / outage.
  • Review-graph centrality (15%) — PageRank (α=0.85) on the directed reviewer→author graph. Whose review do other high-impact people seek?

Composite = Σ weight × z-score. We then min-max normalize to 0–100 for display. The stacked bar on each card shows each signal's weighted contribution to the composite — positive segments to the right, negative (below-average) signals are clipped to zero in the bar but reflected in the score.

Limitations — we don't read code semantics, so "complexity" is approximated by reach + size cap. Outside contributors and very new hires are under-counted. Review centrality is sensitive to who reviews whom; a small clique can inflate each other.
